# Clicker 2.0 - Полное описание логики работы

## Обзор системы

Clicker 2.0 — это многопоточная система автоматизации, которая управляет несколькими виртуальными браузерами для взаимодействия с ИИ-сервисами (Perplexity, ChatGPT, Gemini, Qwen). Каждый бот работает независимо в изолированной среде и выполняет задачи по сбору данных из ИИ-моделей.

## 2. Компоненты системы

### 2.1 Главный контроллер (`main.py`)

**Задачи:**
- Загрузка конфигурации из `config_main.yaml` и `config_bots.yaml`
- Создание анализатора GPU для обработки изображений
- Инициализация виртуальных сред для каждого бота
- Запуск основного цикла обработки

**Алгоритм работы:**
```python
1. Загрузить config_main.yaml и config_bots.yaml
2. Прочитать bots_count из глобальной конфигурации
3. Для каждого бота (0 до bots_count-1):
   a. Создать VirtualBotEnv с индивидуальной конфигурацией
   b. Запустить браузер по URL из config_main.yaml
   c. Создать FSM с сценарием для данного сайта
   d. Подождать 1 секунду до следующего бота
4. Запустить бесконечный цикл:
   a. Для каждого активного бота:
      - Захватить кадр экрана (get_frame_umat)
      - Выполнить шаг FSM (execute_step)
   b. Если все боты остановлены — завершить работу
```

---

### 2.2 Виртуальная среда бота (`scripts/env_bot.py`)

**Компоненты среды:**
1. **Xvfb** — виртуальный дисплей (X Virtual Framebuffer)
2. **Fluxbox** — легковесный оконный менеджер
3. **x11vnc** — VNC-сервер для мониторинга (порт 5900+bot_id)
4. **Chromium** — оптимизированный браузер

**Оптимизации браузера:**
- JavaScript отключён
- Изображения отключены (`blink-settings=imagesEnabled=false`)
- Аппаратное ускорение отключено
- Частота кадров ограничена (`--limit-fps=5`)
- Уведомления отключены
- JavaScript отключён полностью

**Инициализация бота:**
```python
1. Подготовить временный профиль браузера (копия master-профиля)
2. Очистить кэш (Cache, Code Cache, GPUCache, etc.)
3. Удалить файлы блокировок (SingletonLock, SingletonSocket)
4. Запустить Xvfb на дисплее :100+bot_id
5. Запустить Fluxbox
6. Запустить x11vnc на порту 5900+bot_id
7. Запустить Chromium с флагами оптимизации
8. Запустить поток executor для обработки действий
```

**Загрузка вопросов:**
```python
1. Прочитать dataset_path (CSV файл)
2. Использовать row_range для определения диапазона строк
3. Загрузить вопросы с skiprows и nrows для эффективности
4. Инициализировать cur_global_idx = start_row
5. Инициализировать total_question_count = 0
```

### 2.3 Конечный автомат (`scripts/bot_logic.py`)

**Структура FSM:**
```yaml
FSM:
  - current_state: текущее состояние
  - scenario: сценарий из config_main.yaml
  - states: словарь состояний
  - timeout: таймаут для каждого состояния
```

**Структура состояния:**
```yaml
state_name:
  expect:           # Что искать на экране
    templates: [...]
    threshold: 0.8
  action:           # Какое действие выполнить
    - click
    - click_paste_enter
    - click_copy_save_json_check
  condition:        # Условие перехода
    templates: [...]
    json_valid: true
  next:             # Куда перейти
    success: next_state
    fail: other_state
  timeout: 120.0    # Таймаут состояния
```

**Алгоритм execute_step:**
```python
1. Проверить таймаут текущего состояния
   - Если превышен → выполнить reset_scenario
   
2. Если expected_complete = False:
   a. Искать шаблон из expect на экране
   b. Если найден:
      - Установить expected_complete = True
      - Сбросить таймер last_change
      - Очистить буфер обмена (если ожидается JSON)
      - Выполнить действие (_run_action)
   
3. Если expected_complete = True:
   a. Проверить условие перехода:
      - templates: искать шаблон на экране
      - json_valid: верифицировать JSON из буфера
   b. Определить следующее состояние (success/fail)
   c. Сбросить expected_complete = False
   d. Обновить current_state
```

---

### 2.4 Анализатор GPU (`scripts/gpu_analyzer.py`)

**Задачи:**
- Загрузка шаблонов изображений из директории `templates/`
- Сопоставление шаблонов с текущим кадром
- Возврат координат лучшего совпадения

**Алгоритм find_best_match:**
```python
1. Для каждого шаблона:
   a. Загрузить шаблон (кэшировать если возможно)
   b. Применить matchTemplate OpenCV
   c. Найти максимальное значение соответствия
2. Вернуть координаты лучшего совпадения
   - Если max_value >= threshold → успех
   - Иначе → None
```

---

### 2.5 Верификация JSON (`scripts/verification_saved_json.py`)

**Алгоритм проверки:**
```python
1. Прочитать содержимое буфера обмена
2. Найти JSON (первая '{' и последняя '}')
3. Попытаться распарсить JSON
   - Если ошибка → использовать json_repair для восстановления
4. Проверить структуру:
   - Наличие ключа "Norms"
   - Количество норм > 1
5. Очистить данные:
   - clean_micro_art: убрать "ст.", "статья" из art
   - clean_micro_number: убрать "null", "N/A", "б/н"
   - clean_micro_act: сократить названия актов
6. Вернуть (success, data)
```

## 4. Система лимитов вопросов

### 4.1 Конфигурация лимита

```yaml
bot_0:
  max_questions: 100    # Общий лимит на все вопросы
  row_range: [0, 100]   # Диапазон вопросов в CSV
```

### 4.2 Подсчёт попыток

**Важно:** Каждая попытка задать вопрос (успешная или нет) увеличивает глобальный счётчик:

| Событие | total_question_count | Действие |
|---------|---------------------|----------|
| Вопрос #1, JSON валиден | 1 | Сохранить, следующий вопрос |
| Вопрос #2, JSON битый | 2 | Повторить вопрос #2 |
| Вопрос #2, JSON битый | 3 | Повторить вопрос #2 |
| Вопрос #2, JSON валиден | 4 | Сохранить, следующий вопрос |
| ... | ... | ... |
| total_question_count = 100 | 100 | **СТОП** |

### 4.3 Логика проверки

```python
# Перед каждой попыткой задать вопрос
if not bot.check_question_limit():
    return  # Лимит исчерпан, выход

# После каждой попытки (успешной или нет)
if not bot.increment_question_count():
    return  # Лимит превышен, остановка бота
```

## 5. Обработка состояний и переходов

### 5.1 Пример сценария (Perplexity Grok)

```yaml
perplexity:
  scenarios:
    grok:
      start_state: start_question
      
      states:
        start_question:
          expect:
            templates: [ask_anything.png]
            threshold: 0.8
          action: click
          condition:
            templates: [model.png]
            threshold: 0.8
          next:
            success: open_model_menu
            fail: model_selected
          timeout: 120.0
        
        open_model_menu:
          expect:
            templates: [model.png]
            threshold: 0.8
          action: click
          condition:
            templates: [grok_selected.png]
            threshold: 0.7
          next:
            success: ask_empty
            fail: start_question
```

### 5.2 Механизм переходов

```python
# 1. Ожидание визуального триггера
coords = analyzer.find_best_match(frame, expect_templates, threshold)

# 2. Выполнение действия
if coords:
    _run_action(bot, action, coords)

# 3. Проверка условия перехода
if condition.templates:
    success = analyzer.find_best_match(frame, condition_templates)
elif condition.json_valid:
    success = verify_json_from_clipboard()

# 4. Переход к следующему состоянию
next_state = cur_state['next']['success' if success else 'fail']
```

## 8. Остановка бота

### 8.1 Условия остановки

1. **Достигнут последний вопрос в row_range**
   ```python
   if cur_global_idx > end_row:
       stop_event.set()
   ```

2. **Исчерпан лимит max_questions**
   ```python
   if total_question_count >= max_questions:
       stop_event.set()
   ```

3. **Пользователь нажал Ctrl+C**
   ```python
   except KeyboardInterrupt:
       for bot in bots:
           bot.stop()
   ```

### 8.2 Завершение работы

```python
def stop(self):
    stop_event.set()  # Сигнал остановки
    
    # Завершить все процессы
    for proc in [xvfb, wm, vnc, browser]:
        proc.terminate()
    
    # Очистить временный профиль
    shutil.rmtree(temp_profile)
```

## 15. Логирование событий

### 15.1 Структура логов

Каждый бот записывает события в индивидуальный лог-файл:
```
logs/
└── bot_events/
    ├── bot_0_events.log
    ├── bot_1_events.log
    └── ...
```

### 15.2 Формат записи

Каждая запись в логе имеет формат:
```
[YYYY-MM-DD HH:MM:SS] [TYPE] Сообщение
```

Типы событий:
- **START** — Запуск бота (индекс вопроса, всего вопросов задано)
- **RESTART** — Перезапуск бота (время с последнего запуска)
- **QUESTION** — Попытка задать вопрос (номер попытки, индекс вопроса, UID)
- **LIMIT_EXHAUSTED** — Лимит вопросов исчерпан (всего задано, время до перезапуска)
- **STOP** — Бот остановлен (всего вопросов задано)

### 15.3 Пример лога

```
[2026-03-07 10:00:00] [RESTART] Бот перезапущен (первый запуск)
[2026-03-07 10:00:05] [QUESTION] Попытка #1/100, вопрос индекс #527, UID: 12345
[2026-03-07 10:00:10] [QUESTION] Попытка #2/100, вопрос индекс #527, UID: 12345
[2026-03-07 10:00:15] [QUESTION] Попытка #3/100, вопрос индекс #528, UID: 12346
...
[2026-03-07 13:00:00] [LIMIT_EXHAUSTED] Лимит вопросов исчерпан. Всего задано: 100. Перезапуск через 3600 сек
[2026-03-07 13:00:00] [STOP] Бот остановлен. Всего вопросов задано: 100
[2026-03-07 14:00:00] [RESTART] Бот перезапущен. Прошло времени с последнего запуска: 1:00:00
```

### 15.4 Логирование в коде

```python
# В env_bot.py
def increment_question_count(self):
    self.total_question_count += 1
    uid = self.get_cur_question_uid()
    self.log_question_attempt(self.cur_global_idx, uid)  # Логирование
    
def log_question_attempt(self, question_idx: int, question_uid: str = None):
    """Логирует каждую попытку вопроса"""
    uid_info = f", UID: {question_uid}" if question_uid else ""
    self._log_event("QUESTION", f"Попытка #{self.total_question_count}/{self.max_questions}, вопрос индекс #{question_idx}{uid_info}")
```

## 16. Автоматический перезапуск ботов

### 16.1 Конфигурация перезапуска

```yaml
bot_0:
  max_questions: 100       # Лимит вопросов
  restart_delay: 3600      # Задержка перед перезапуском (секунды)
```

**Важно:** `restart_delay` отсчитывается от момента **последнего запуска** бота, а не от момента остановки.

Пример:
- 10:00:00 — Бот запущен
- 13:00:00 — Лимит исчерпан (бот работал 3 часа)
- 14:00:00 — Перезапуск (ровно через 1 час от момента запуска, а не от остановки)

### 16.2 Алгоритм перезапуска

```python
# В main.py
while True:
    # Проверка остановленных ботов
    for bot_idx, (bot, fsm) in enumerate(zip(bots, logics)):
        if bot.stop_event.is_set():
            restart_delay = bot_cfg.get('restart_delay', 0)
            
            if restart_delay > 0:
                # Запоминаем время остановки
                if bot_idx not in bot_stop_times:
                    bot_stop_times[bot_idx] = current_time
                
                # Проверяем, прошло ли достаточно времени
                elapsed = current_time - bot_stop_times[bot_idx]
                if elapsed >= restart_delay:
                    # Перезапускаем бота
                    bot.stop()
                    new_bot = VirtualBotEnv(bot_idx, bot_cfg)
                    new_bot.start(url)
                    bots[bot_idx] = new_bot
```

### 16.3 Временная шкала перезапуска

```
10:00:00 — Бот запущен (START)
10:00:05 — Вопрос #1 (успех)
10:00:10 — Вопрос #2 (битый JSON, повтор)
10:00:15 — Вопрос #2 (успех)
...
13:00:00 — Вопрос #100 (лимит исчерпан, LIMIT_EXHAUSTED)
13:00:00 — Бот остановлен (STOP)
13:00:00 — Ожидание: следующий перезапуск в 14:00:00
14:00:00 — Перезапуск (RESTART) — ровно через 1 час от СТАРТА
14:00:05 — Вопрос #101 (начало нового цикла)
```

**Ключевой момент:** Бот перезапускается в `14:00:00`, что равно `10:00:00 + 3600 сек`, а не `13:00:00 + 3600 сек = 14:00:00`.

### 16.4 Отключение перезапуска

```yaml
# Бот не будет перезапущен автоматически
bot_0:
  max_questions: 100
  restart_delay: 0  # Перезапуск отключён
```

## Заключение

Clicker 2.0 — это масштабируемая система автоматизации с:
- **Независимыми ботами** — каждый работает в изолированной среде
- **FSM логикой** — гибкое управление сценариями
- **Верификацией JSON** — гарантия качества данных
- **Глобальным лимитом** — контроль количества попыток
- **Логированием событий** — полная история работы каждого бота
- **Автоматическим перезапуском** — продолжение работы после исчерпания лимита
- **Оптимизацией** — эффективное использование ресурсов

Система предназначена для автоматизированного сбора данных из ИИ-сервисов с возможностью масштабирования до 23+ одновременных ботов.
